# 案例：缺料停线

## 问题现象

L01 产线进入危机状态（crisis 场景），**库存仅剩 4 小时**（`inventoryHours=4`，红线 8h），**缺料风险高达 78%**，已发生 1 次物料缺料待机（180min），OEE 跌至 34%，可用率 55%。

## 诊断过程

1. `material.wip_level`：WIP 水位 880（超红线 600，黄线 450），但**全部堆积在瓶颈工序前**，瓶颈后缺料
2. `material.supply_risk`：缺料风险 0.78（极高）
3. `material.inventory`：库存仅 4h，远低于 8h 安全线
4. `equipment.downtime`：停机事件含"物料缺料待机 180min"
5. `schedule.capacity`：产能利用率 1.15（过载），加剧物料消耗速度

## 根因

| 原因层 | 分析 |
|--------|------|
| 直接原因 | 瓶颈工序后缺料，导致下游工序停机待料 |
| 流程原因 | WIP 堆积在上游（虚假的"库存充足"），实际有效供给不足 |
| 系统原因 | 排产未与物料供应联动（`schedule.capacity` 过载但未触发补料预警） |

**5Why**：
- Why1 为什么停线？→ 缺料
- Why2 为什么缺料？→ 库存仅 4h
- Why3 为什么库存低？→ 排产过载消耗快 + 未及时补料
- Why4 为什么没及时补料？→ 无库存低于 8h 的自动预警机制
- Why5 为什么无预警？→ 物料与排产系统未联动

**根因**：物料系统与排产系统割裂，缺料预警滞后。

## 对策

### 临时（止损）
- `mcp.erp.material_issue`：紧急领料出库至 L01 线边
- `mcp.erp.purchase_request`：紧急采购补货（urgency=urgent）

### 永久（系统改善）
1. **建立库存预警联动**：`material.inventory` < 8h 时自动触发 `schedule` 限产建议
2. **WIP 结构治理**：不是总量够，是分布错 → 拉动式补料（看板），而非推动式堆料
3. **排产前物料校验**：排产（`mcp.mes.schedule_work_order`）前强制检查 `material.supply_risk`，高风险时拒绝加单

## 教训沉淀

- **WIP 总量高 ≠ 不缺料**：关键看瓶颈后的有效供给，而非上游堆积
- **库存 8h 是硬红线**：低于此值必须立即行动，不能等停线
- **物料与排产必须联动**：两个系统割裂是缺料停线的系统性根因

## 适用工具调用模式

```
material.wip_level → material.supply_risk → material.inventory → equipment.downtime
↓ (对策)
mcp.erp.material_issue → mcp.erp.purchase_request → material.inventory (复检)
```
