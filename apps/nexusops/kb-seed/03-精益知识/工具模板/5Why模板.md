# 5Why 分析法模板

5Why（五个为什么）由丰田佐吉提出、大野耐一推广，是丰田生产系统（TPS）根因分析的核心工具。通过对一个问题连续追问"为什么"，沿因果链追溯根本原因，而非停留在表面症状修补。

## 核心原则

1. **基于事实与数据**：每个"为什么"的回答必须有现场证据支撑，禁止臆测或归咎于人
2. **因果必然**：相邻两层之间因果必须逻辑必然，不跳跃、不发散
3. **三现主义**：到现场、看现物、把握现实（Genchi Genbutsu）
4. **问到可控层**：当追到"可通过具体行动改变的流程/设计层"即可停止，**不必强求正好 5 次**（可能 3 次，可能 7 次）
5. **对策 ≠ 修补**：对策应杜绝再发（系统预防），而非仅消除当前症状

## 三层面分析框架

完整的 5Why 应从三个层面分别追问，缺一不可：

| 层面 | 追问方向 | 目的 |
|------|----------|------|
| 制造层（为什么会发生） | 直接导致问题的工艺/操作原因 | 找出技术根因 |
| 检验层（为什么没发现） | 为什么缺陷流出到下一工序/客户 | 找出探测漏洞 |
| 体系层（为什么没预防） | 为什么流程/制度没拦住 | 找出系统性漏洞 |

> 只有三个层面都探到底，才不会"修了表面，根还在"。

## 标准模板

```markdown
## 问题描述
- 现象：[一句话描述可见症状]
- 时间：[发生时间]
- 地点：[产线/工位]
- 影响：[量化损失，如报废 XX 件 / 停机 XX min]

## 制造层追问（为什么会发生）
- Why1：为什么 [现象]？→ [答1（直接原因）]
- Why2：为什么 [答1]？→ [答2]
- Why3：为什么 [答2]？→ [答3]
- Why4：为什么 [答3]？→ [答4]
- Why5：为什么 [答4]？→ [答5（根本原因）]

## 检验层追问（为什么没发现）
- Why1：为什么检验没拦住？→ [答]
- ...（追问至探测漏洞根因）

## 体系层追问（为什么没预防）
- Why1：为什么 SOP/制度没防住？→ [答]
- ...（追问至系统漏洞根因）

## 根本原因结论
[综合三层，提炼真正的根因]

## 对策
- 临时遏制：[止损动作，立即执行]
- 永久预防：[针对根因的系统性措施，如改 SOP / 加防错 / 改设计]
```

## 经典案例：丰田机器停机

| 层次 | 问 | 答 |
|------|----|----|
| Why1 | 为什么机器停了？ | 机器超载，保险丝烧断 |
| Why2 | 为什么超载？ | 轴承润滑不足 |
| Why3 | 为什么润滑不足？ | 润滑泵失灵 |
| Why4 | 为什么泵失灵？ | 泵轮轴磨损 |
| Why5 | 为什么磨损？ | 没装过滤器，杂质混入 |

- **错误对策**：换保险丝（只修症状，几小时后还会停）
- **正确对策**：安装过滤器 + 建立过滤器定期更换制度（根因预防）

## 停止判定标准

满足以下任一条件可停止追问：

1. 已追到**可通过具体行动改变**的流程/设计层（如"没有过滤器"）
2. 再往下问会进入**无法控制**的因素（如"供应商倒闭""经济环境"）
3. 当前层的对策已能**系统预防再发**

## 与 NexusOps 工具配合

| 阶段 | 工具 | 用途 |
|------|------|------|
| 描述问题 | `oee.realtime` / `quality.defect_rate` | 取量化现象数据 |
| Why1-2 定位症状层 | `quality.pareto` / `equipment.health` | 锁定主导缺陷/异常设备 |
| Why3-4 定位根因层 | `quality.root_cause_5m1e` | 5M1E 脚手架辅助深挖 |
| 对策验证 | `process.parameters`（调参后复检） | 验证对策是否生效 |

## 常见误区

- **归咎于人**：答到"员工粗心"就停 → 应继续问"为什么员工会粗心"（培训？SOP 模糊？疲劳？）
- **跳跃逻辑**：Why2 跳到 Why5 的结论，中间因果不必然
- **只做制造层**：忽略检验层/体系层，问题换种形式还会复发
