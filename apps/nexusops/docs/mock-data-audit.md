# NexusOps 数据资产审查报告

> 审查范围：`apps/nexusops/tools/mock-data/*`、`apps/nexusops/tools/domains/*`、`data/nexus-vault/**/*`、`data/nexus-eval/**/*`
> 审查目标：回答三个问题——① 现有 mock/基础数据哪些是常规软件能取到的；② 哪些是系统元数据、哪些是查询组合出的；③ 因果链/推理链数据的格式、来源与积累机制

---

## 一、核心结论（TL;DR）

| 维度 | 结论 |
|------|------|
| **常规软件能否取到** | 约 **85%** 的 mock 数据是常规 MES/MOM/ERP/PLM/EAM 系统的标准数据，真实部署后可直接对接；约 **15%** 是"分析产物"或"管理沉淀"，需要额外建模或人工录入 |
| **系统元数据 vs 查询组合** | 系统元数据（实测快照）占 **~70%**，查询组合数据（派生/聚合/比对）占 **~30%**；后者纯由前者经代码派生，无需独立数据源 |
| **因果链数据本质** | 当前是**硬编码的人工沉淀**（scenario 级固化），**不是实时推理产物**；这是 NexusOps 最具特色也最值得深挖的数据类别 |
| **因果链积累路径** | 三阶段：当前硬编码 → 中期"案例库 + 症状匹配"半自动 → 长期"轨迹挖矿 + 验证沉淀"全自动（与 skill-miner 机制呼应） |

## 二、数据全景分类

### 2.1 三类数据的定义

```
┌─────────────────────────────────────────────────────────────────────┐
│                  NexusOps 数据资产三层模型                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Layer 1: 系统元数据（实测快照）  ← 常规软件直接取                   │
│  ─────────────────────────────────                                  │
│  MES/MOM/ERP/PLM/EAM 原生数据，单点查询即得                          │
│  例：OEE 数值、设备状态、缺陷计数、工艺参数实测值、库存水位           │
│                                                                     │
│  Layer 2: 查询组合数据（派生/聚合/比对）  ← 代码派生                 │
│  ──────────────────────────────────────                              │
│  基于多个 Layer 1 字段计算得出，工具的 getData() 函数产出            │
│  例：损失瀑布、瓶颈排名、趋势环比、标准 vs 实测对照、动作副作用      │
│                                                                     │
│  Layer 3: 因果链/推理数据（沉淀的领域知识）  ← 当前硬编码            │
│  ──────────────────────────────────────────                          │
│  带有"为什么"的解释性数据，跨域关联，需要人工或算法沉淀              │
│  例：5Why 链、鱼骨图 5M1E 分支、根因判定、案例库索引                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 按"常规软件能否取到"分类

| 能取到（常规系统） | 系统来源 | 不能直接取（需额外建模/沉淀） |
|--------------------|----------|-------------------------------|
| OEE 数值 + 三项分解 | MES | **因果链（5Why layers）** |
| 设备状态/MTBF/MTTR | MES/PLM | **鱼骨图 5M1E 分支带证据** |
| 停机事件清单 | MES | **PFMEA 的 S/O/D 评分**（部分需专家评定） |
| 缺陷率/帕累托/SPC | MOM | **失败预测 30 天风险**（需 ML 或经验公式） |
| 工艺参数实测值 | MES | **改善阻力记录/人员态度**（管理类，非系统数据） |
| 工艺配方（标准） | PLM | **A3 案例库**（事后沉淀） |
| 能耗实时/单耗 | EMS/SCADA | **PDCA 阶段判定**（推理辅助） |
| 库存/WIP 水位 | ERP/MES | **CT vs Takt 诊断规则**（管理标准） |
| 排产达成率 | MES | **Cpk 不足时的根因链**（需诊断流程产出） |
| 人员技能等级 | HRM/培训系统 | **班次差异的"为什么"**（需交叉分析） |
| 报废成本 | ERP | **约束条件模板**（决策边界，管理定义） |

> **关键洞察**：常规系统能取到的是"是什么"（What），取不到的是"为什么"（Why）和"该怎么办"（How）。NexusOps 的差异化价值正是在 Layer 3。

## 三、系统元数据（Layer 1）详细审查

### 3.1 这类数据的特征

1. **单点查询即得**：一次 API/SQL 调用返回，不需跨表 JOIN
2. **time-stamped**：带 `capturedAt`，有时效性（realtime/shift/daily）
3. **confidence=measured**：实测数据，非推断
4. **来源系统明确**：每个工具的 `system` 字段标注 MES/MOM/ERP/PLM/EAM

### 3.2 现有 Layer 1 数据清单

| 数据簇 | mock 位置 | 真实系统 | 常规软件能力评估 |
|--------|-----------|----------|------------------|
| **OEE 基础** | `OEE_BASE` | MES | ✅ 标配，所有 MES 都有 OEE 模块 |
| **设备数据** | `EQUIPMENT` | MES+PLM | ✅ EAM/SCADA 标配，健康分需额外算法 |
| **质量数据** | `QUALITY` | MOM/QMS | ✅ 缺陷计数/SPC 标配；Cpk 由 SPC 模块算 |
| **工艺参数** | `PROCESS.parameters` | MES+PLM | ✅ 实测值标配；**S/O/D 评分需 PFMEA 评定** |
| **能耗** | `ENERGY` | EMS/IoT | ✅ 实时功率标配；碳排系数需配置 |
| **排产** | `SCHEDULE` | MES/APS | ✅ 达成率标配；瓶颈识别 APS 才有 |
| **物料** | `MATERIAL` | ERP+MES | ✅ WIP/库存标配 |
| **人员技能** | `PERSONNEL` | HRM | ⚠️ 技能等级需培训系统维护，非所有企业有 |
| **班次偏移** | `SHIFT_DEVIATION` | MES | ✅ 按班次聚合是标配能力 |

### 3.3 Layer 1 小结

**约 90% 是常规软件标配**，真实部署后可直接对接 MES/MOM/ERP。唯一需额外建模的是：
- `healthScore`（设备健康分）—— 需振动/温度算法或经验阈值
- `failureRisk30d`（30 天故障风险）—— 需预测性维护 ML 模型
- PFMEA 的 `severity/occurrence/detection` —— 需工艺工程师评定

## 四、查询组合数据（Layer 2）详细审查

### 4.1 这类数据的特征

1. **代码派生**：`getData()` 函数消费 Layer 1 字段计算
2. **零独立存储**：不占 mock seed，运行时算出
3. **确定性可复现**：相同输入永远相同输出
4. **仍是 measured 置信度**（因为底层数据是实测的）

### 4.2 现有 Layer 2 数据清单（按派生方式分类）

#### A. 聚合/排序类

| 工具 | 派生逻辑 | 输入来源 |
|------|----------|----------|
| `oee.decompose` | 把 OEE 拆成三项损失瀑布，找最大损失项 | `OEE_BASE` |
| `oee.bottleneck` | 多产线 OEE 排序取最低 | `OEE_BASE`（全产线） |
| `oee.by_shift` | 基准 OEE + 班次偏移 | `OEE_BASE` + `SHIFT_DEVIATION` |
| `equipment.downtime` | 汇总停机事件总时长 + topReason | `EQUIPMENT.downtimeEvents` |
| `quality.pareto` | 缺陷类型按 count 排序（mock 已预排序） | `QUALITY.topDefects` |

#### B. 比对/差异类

| 工具 | 派生逻辑 | 输入来源 |
|------|----------|----------|
| `oee.trend` | 本周 vs 上周 OEE 环比 | `OEE_BASE.trend7d` |
| `oee.compare` | 两产线 OEE 差值 | `OEE_BASE`（双产线） |
| `process.deviation` | 实测 vs 标准，算 deltaPct | `PROCESS.parameters` |
| `process.standard_vs_actual` | 标准/实测/公差三列对照 | `PROCESS.parameters` |

#### C. 派生指标类

| 工具 | 派生逻辑 | 输入来源 |
|------|----------|----------|
| `process.fmea` | 从参数的 S/O/D 算 AP 行动优先级 | `PROCESS.parameters` + `computeAP()` |
| `quality.spc` | 由 cpk 判定是否受控 + 生成控制限 | `QUALITY.cpk` |
| `quality.cp_cpk` | cpk → adequate/marginal/inadequate 评级 | `QUALITY.cpk` |
| `quality.scrap` | scrapRate × 1000 × 45 元 折算报废成本 | `QUALITY.scrapRate` |
| `oee.report_html` | 多域数据组装成 HTML 报告 | OEE+设备+质量+工艺+因果链 |

#### D. 动作副作用类（特色）

| 工具 | 派生逻辑 | 输入来源 |
|------|----------|----------|
| `applyOeeOverrides()` | 停线 → 可用率归零；调参 → 性能率回升 | `actionStore` overrides |
| `applyQualityOverrides()` | 隔离/报废/返工 → 缺陷率下降 | `actionStore` overrides |
| `applyMaterialOverrides()` | 领料/采购 → 库存回升/风险下降 | `actionStore` overrides |

> **这是 NexusOps 最巧妙的设计**：动作工具（write/destructive）执行后写入 `actionStore`，读取工具通过 `lookupActionOverride` 消费，形成"执行→复检"闭环。这部分**真实部署时需对接业务系统的状态变更回调**，不是简单查询。

### 4.3 Layer 2 小结

**Layer 2 占工具总数的 ~30%，但零数据存储负担**——全部由 Layer 1 经代码派生。真实部署时，只要 Layer 1 数据接通，Layer 2 工具**几乎无需改动**（除动作副作用需对接业务系统 webhook 外）。

## 五、因果链/推理链数据（Layer 3）深度剖析

### 5.1 这类数据的特殊性

因果链数据是 NexusOps **最具差异化也最值得深挖**的数据类别。它与其他两层有本质区别：

| 维度 | Layer 1（实测） | Layer 2（派生） | **Layer 3（因果链）** |
|------|-----------------|-----------------|----------------------|
| 回答的问题 | 是什么（What） | 有多糟（How bad） | **为什么（Why）** |
| 数据来源 | 传感器/业务系统 | Layer 1 经计算 | **专家经验 + 历史案例** |
| 置信度 | measured | measured | **inferred**（标注推断） |
| 可复现性 | 完全可复现 | 完全可复现 | **同一场景可复现，跨场景需重沉淀** |
| 真实软件能否取 | ✅ 标配 | ✅ 派生即得 | ❌ **需自建沉淀机制** |

### 5.2 因果链数据的格式

当前因果链数据硬编码在 `scenarios.ts` 的 `CAUSAL_CHAIN` 常量，结构如下：

```typescript
interface CausalChainData {
  symptom: string;        // 表层症状（与 quality.defectRate / oee 等输出对齐）
  chains: Array<{
    method: "5why";
    layers: string[];     // 5Why 逐层追问链（现象→直接原因→…→根本原因）
    rootCause: string;    // 根本原因
  }>;
  fishbone: {             // 5M1E 六分支，每分支带证据引用
    man: string[];        // 人——证据指向 PERSONNEL / SHIFT_DEVIATION
    machine: string[];    // 机——证据指向 EQUIPMENT.healthScore / mtbfHours
    material: string[];   // 料——证据指向 MATERIAL
    method: string[];     // 法——证据指向 PROCESS.parameters
    environment: string[];
    measurement: string[];
  };
}
```

#### 5.2.1 格式解读（以 anomaly.L01 为例）

```13:42:apps/nexusops/tools/mock-data/scenarios.ts
  symptom: "L01 尺寸超差率 5.8%，Cpk 0.85 < 1.0（能力不足）",
```

**关键设计特征**：

1. **symptom 与工具输出对齐**：症状描述引用了 `quality.defect_rate`（5.8%）和 `quality.cp_cpk`（0.85）的具体数值，不是空泛描述。这让 LLM 在 ReAct 时能把症状与取证数据关联。

2. **5Why 链是因果递进，非简单罗列**：

```468:477:apps/nexusops/tools/mock-data/scenarios.ts
            "现象：尺寸超差率 5.8%，主缺陷类型为'尺寸超差'（占 42%）",
            "为何尺寸超差？主轴径向跳动 0.03mm 超规（标准 ≤0.02mm）",
            "为何主轴跳动超规？主轴前轴承磨损（间隙增大）",
            "为何轴承磨损加速？自动润滑系统供油不足",
            "根本原因：自动润滑泵滤网堵塞，导致供油不足 → 轴承异常磨损 → 主轴跳动 → 尺寸超差",
```

每层是"上一层现象的下一层原因"，直到到达**物理根因**（滤网堵塞——可物理干预的最终环节）。

3. **鱼骨图证据指向具体 mock 字段**：

```479:486:apps/nexusops/tools/mock-data/scenarios.ts
        man: ["C 班夜班缺陷率比 A 班高 0.012（见 SHIFT_DEVIATION.L01.C）", "C 班含新员工未独立（见 PERSONNEL 班长B 郑师傅 level）"],
        machine: ["主轴健康分 0.62 < 0.7 阈值（见 EQUIPMENT.L01.healthScore）", "MTBF 降至 180h（正常 480h，见 EQUIPMENT.L01.mtbfHours）", "停机事件 3 起：模具卡死/传感器漂移/换模超时（见 EQUIPMENT.L01.downtimeEvents）"],
```

每条证据**显式引用 mock 字段路径**（如 `EQUIPMENT.L01.healthScore`），而非"设备有问题"这种空泛描述。这是**证据可追溯性**的关键设计。

### 5.3 因果链数据应该从何而来（来源分析）

当前硬编码方式适合 demo/测试，但真实部署时**绝不能依赖 LLM 实时推理**（不可控、不可复现、易幻觉）。因果链数据应有三个来源：

#### 来源 1：历史案例库（主来源，结构化沉淀）

`data/nexus-vault/02-改善项目/A3报告库/` 已有 6 个 A3 案例，每个都是一次真实诊断的沉淀：

```
data/nexus-vault/02-改善项目/A3报告库/
├── 主轴轴承异响案例.md      ← 设备致质量
├── 换模超时致可用率损失案例.md ← SMED
├── Cpk不足致尺寸超差案例.md   ← 工艺致质量
├── 缺料停线案例.md           ← 物料致OEE
├── 能耗飙升诊断案例.md       ← 能耗异常
└── WIP堆积致物料流堵塞案例.md ← 物料流
```

这些 A3 案例的标准结构（问题现象→诊断过程→根因→对策→教训）**天然就是因果链的雏形**。目前是 Markdown 自由文本，未来应结构化为可被症状匹配的因果链模板。

**案例索引**（`03-精益知识/案例库/设备致质量案例索引.md`）已做了按症状检索的雏形，把"症状 → 案例 → 诊断路径 → 根因"的映射显式化了。

#### 来源 2：专家规则/诊断标准（辅来源，知识库化）

`data/nexus-vault/` 已沉淀了大量诊断规则，这些是"半结构化的因果知识"：

| 知识文件 | 因果知识类型 |
|----------|--------------|
| `01-现场状态/OEE计算口径.md` | "OEE 低 → 按最大损失项诊断"的决策树 |
| `01-现场状态/节拍与CT标准.md` | "CT > Takt → 攻关瓶颈"的判定规则 |
| `01-现场状态/WIP水位阈值.md` | "WIP 超红线 → 查瓶颈/查质量"的处置流程 |
| `01-现场状态/设备台账与关键备件.md` | "备件=0 → 临时方案+加急采购"的约束 |
| `05-推理辅助/约束条件模板.md` | 决策硬约束（安全红线/资源约束/证据强度） |
| `05-推理辅助/PDCA阶段判定.md` | "意图关键词 → PDCA 阶段 → 工具选择"映射 |
| `04-人与组织/培训与技能矩阵.md` | "班次缺陷差异>5% → 查技能等级"的关联规则 |

这些规则**可编译成症状→根因的映射表**，成为因果链的"规则引擎"层。

#### 来源 3：轨迹挖矿（长期来源，自动化）

参考项目已有的 `skill-miner.ts` 机制（见 [17-skill-sedimentation.md]），已验证的 ReAct 轨迹可自动挖掘为 skill。同理，**已验证的诊断轨迹（症状→取证序列→根因→对策有效）可挖矿为因果链**。

信号设计（与 skill-miner 呼应）：
- **硬信号**：同一症状簇 ≥3 次诊断 + 最终根因被现场确认 + 对策有效（指标回升）
- **反信号一票否决**：根因是 inferred 硬结论 / HITL 否决 / 对策无效

### 5.4 因果链数据的积累机制（三阶段演进）

```
┌──────────────────────────────────────────────────────────────────────┐
│                  因果链数据积累的三阶段演进                            │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  阶段 1（当前）：硬编码 Scenario 级                                   │
│  ─────────────────────────────────────                                │
│  • CAUSAL_CHAIN 按 (scenarioId, line) 固化                           │
│  • 适合 demo/e2e 测试，保证可复现                                     │
│  • 缺点：覆盖面窄（仅 anomaly/crisis 的 L01/L02 有链），不可泛化      │
│                                                                      │
│  阶段 2（中期）：案例库 + 症状匹配（半自动）                          │
│  ─────────────────────────────────────────────                        │
│  • A3 案例库结构化为 {symptom_pattern, chains, fishbone}              │
│  • 工具 quality.five_why 改为：读实时症状 → 匹配案例库 → 返回链       │
│  • LLM 仍可补充，但优先用历史沉淀                                     │
│  • 新案例由用户/工程师审核后入库                                      │
│                                                                      │
│  阶段 3（长期）：轨迹挖矿 + 验证沉淀（全自动）                        │
│  ─────────────────────────────────────                                │
│  • 复用 skill-miner 机制：诊断轨迹 → 候选因果链 → draft 影子 → 转正   │
│  • "诊断对了吗"由后续指标变化验证（对策执行后 OEE 回升=根因对的）      │
│  • 持续积累企业专属的因果知识图谱                                     │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

#### 5.4.1 阶段 2 的落地建议（中期最值得做）

当前 `quality.five_why` 和 `quality.fishbone` 直接读 `CAUSAL_CHAIN` 硬编码。建议改造为：

```
用户问"L01 为什么 OEE 低"
  ↓
1. oee.realtime 取实测 → 症状 = "OEE=0.61, 可用率=0.78, 质量率=0.92"
  ↓
2. quality.five_why 调用：
   a. 先按症状匹配案例库（症状相似度 > 阈值）
   b. 命中 → 返回历史因果链（confidence=high, 标注"历史案例匹配"）
   c. 未命中 → 返回 LLM 推理的链（confidence=inferred, 标注"实时推理待验证"）
  ↓
3. 诊断完成后，引导用户"是否沉淀为新案例"
```

这样既保留 demo 的可复现性，又开启了真实积累通道。

## 六、风险与改进建议

### 6.1 已识别的风险

| 风险 | 严重度 | 说明 |
|------|--------|------|
| **R1: 因果链硬编码导致泛化弱** | 中 | 真实场景的症状千变万化，3 个 scenario × 3 产线远远不够 |
| **R2: 因果链与实测数据可能脱节** | 中 | 硬编码的 symptom 写死"5.8%"，若实测变化则不一致（demo 无妨，真实部署有问题） |
| **R3: PFMEA 的 S/O/D 评分缺乏来源** | 低 | 当前硬编码，真实需工艺工程师评定并维护 |
| **R4: 人员技能矩阵维护成本** | 低 | 需 HRM/培训系统持续更新，非所有企业有 |
| **R5: 动作副作用是 mock 内存** | 低 | 真实部署需对接业务系统状态变更回调 |

### 6.2 改进建议（按优先级）

#### P0（真实部署前必做）

1. **因果链解耦**：把 `CAUSAL_CHAIN` 从硬编码改为"症状匹配 + 案例库查询"，symptom 字段改为运行时从实测数据组装（而非硬编码数值）
2. **动作副作用对接**：`actionStore` 的 overrides 改为订阅业务系统 webhook（而非内存 Map）

#### P1（中期增强）

3. **A3 案例结构化**：把 `data/nexus-vault/02-改善项目/A3报告库/*.md` 解析为结构化 JSON（含 symptom_pattern / chains / fishbone / rootCause），供症状匹配检索
4. **因果链置信度分级**：历史案例匹配 = high；LLM 实时推理 = inferred；标注来源让前端可区分展示
5. **PFMEA 评分接入**：S/O/D 从 PLM/FMEA 系统读取，而非硬编码

#### P2（长期演进）

6. **轨迹挖矿沉淀因果链**：复用 skill-miner 机制，把已验证的诊断轨迹自动转化为候选因果链
7. **因果知识图谱**：跨案例关联根因（如"轴承磨损"出现在多个案例 → 构建设备失效模式图谱）

## 八、结语

NexusOps 的数据架构遵循了**清晰的分层纪律**：

- **Layer 1（实测）** 占大头且常规软件能取 —— 解决"接地气"问题
- **Layer 2（派生）** 纯代码零存储 —— 解决"不重复存"问题
- **Layer 3（因果）** 硬编码但可演进 —— 解决"差异化价值"问题

最大的改进空间在 **Layer 3 的积累机制**：从当前的硬编码 demo，演进为"案例库 + 症状匹配 + 轨迹挖矿"的三段式积累。这正是 NexusOps 从"演示工具"走向"企业专属诊断大脑"的关键路径。

建议优先推进 **P0 的因果链解耦** 和 **P1 的 A3 案例结构化**，这两项能最大化复用已有的 `nexus-vault` 知识资产，让 mock 数据真正"活起来"。
